xh2 Expression

Statements are made up of one or more expressions.

An expression is any reference to a variable or value, or a set of variable(s) and value(s) combined with operators. <- OSNOVE!

For example:

a = b * 2;

This statement has four expressions in it:

• 2 is a literal value expression.

• b is a variable expression, which means to retrieve its current value.

• b * 2 is an arithmetic expression, which means to do the multiplication.

• a = b * 2 is an assignment expression, which means to assign the result of the b * 2 expression to the variable a.


Statement sama po sebi ne vraca vrednost dok expression vraca. <- SUSTINA!

xh2 Inheritance

Conflict resolution strategies

Rules for how implementation inheritance work:

  1. Classes win over interfaces. If a class in the superclass chain has a declaration for the method (concrete or abstract), you're done, and defaults (from interface) are irrelevant.
  2. More specific interfaces win over less specific ones (where specificity means "subtyping"). A default from List wins over a default from Collection, regardless of where or how or how many times List and Collection enter the inheritance graph.
  3. If there is not a unique winner according to the above rules, concrete classes must disambiguate manually.

Ovo su pravila koja uvek treba da imamo na umu kada zakljucujemo koji metod ce se naslediti u odredjenom slucaju (kada klasa implementira interface sa npr. default-nim metodama i ekstenduje (apstraktnu) klasu). <- VAZNO.


Par napomena vezaih za default-ne metode u interface-u:

  • default-ni metodi ne mogu override-ovati metode iz Object klase.
  • kada se eksplicitno poziva default-ni metod iz interface-a, koristi se super keyword u obliku: InterfaceName.super.defaultMethodName();. Moze se pozivati samo interface koji klasa direktno implementira, ne i grandparent interface.

    • Primer koda Examples/defaultMethods u Test environment folderu.

  • ne mogu se koristiti final i synchronized keywords uz default-ne metode.

We can inherit static methods.


We call the declared type of the variable the static type, because it is declared in the source code—the static representation of the program.

We call the type of the object stored in a variable the dynamic type, because it depends on assignments at runtime—the dynamic behavior of the program.

Vehicle v1 = new Car();

The static type of v1 is Vehicle, the dynamic type of v1 is Car.

For type checking, the static type is used. The dynamic type is often only known at runtime, so the compiler has no other choice but to use the static type if it wants to do any checks at compile time.

xh2 Compile-time (compiletime) dependency

Compiletime dependencies are only the dependencies (other classes) which you use DIRECTLY in the class you're compiling.

Compiletime dependencies are dependencies that we need on our CLASSPATH in order to compile artifact.


Java doesn't actually link anything at compile time. It only verifies the syntax using the matching classes it finds in the CLASSPATH. It's not until runtime that everything gets put together and executed based on the CLASSPATH at that time.

xh2 Runtime dependency

Runtime dependencies covers both the DIRECT and INDIRECT dependencies of the class you're running.

Runtime dependencies are dependencies that we need on our CLASSPATH in order to run artifact.

Thus, runtime dependencies includes dependencies of dependencies and any reflection dependencies like classnames which you have in a String, but are used in Class#forName().￼

Runtime dependency is more related with the actual flow-of-control.

xh2 Polymorphism ("many shapes")

Property of an object to take on many different forms.

To put this more precisely, a Java object may be accessed using a reference with the same type as the object, a reference that is a superclass of the object, or a reference that defines an interface the object implements, either directly or through a superclass. So reference type and actual object type can be different. This is why variables holding object types in Java are polymorphic variable. Furthermore, a cast is not required if the object is being reassigned to a super type or interface of the object.

So, a variable can hold objects of its declared type or of any subtype of its declared type. This principle is known as substitution - we can substitute a subclass object where a superclass object is expected, because the subclass object is a special case of the superclass.

Superclass se cesto naziva i supertype ili base class dok se subclass cesto naziva subtype ili derived class.


Kompajler koristi reference type da bi odredio koji metodi se mogu pozivati nad objektom dok sa 2. strane JVM koristi object type da bi odlucio koji tacno metod da pozove. JVM ce uvek pozvati najspecificniju verziju metoda, pocevsi od object type-a i iduci ka parent objektima: <- OBRATI PAZNJU. MNOGO VAZNO.

Method binding (method lookup or method dispatch)

Rekli smo da kompajler koristi reference type da bi odredio koji metodi mogu da se pozovu nad odredjenom referencom, dok JVM koristi object type da bi odredio koju implementaciju tog metoda treba pozvati. Ovaj proces se zove method binding. Pored instance metoda, method binding se vrsi i za staticke metode, tj. za svaki metod. Definicija method binding-a je: povezivanje method call-a sa odgovarajucom deklaracijom metoda kao i sa odgovarajucom implementacijom te deklaracije. Method binding se deli na 2 dela:

  1. method signature binding i
  2. method implementation binding.

Method signature binding se sastoji od sledeceg:

  1. Kada kompajler naidje na method invocation statement, on proverava da li tip reference, nad kojom se metod poziva, sadrzi kompatibilan metod, tj. da li postoji metod koji ima iste ili kompatibilne parametre.
  2. Ako kompajler pronadje takav metod onda on upisuje signature tog metoda u bytecode. Ako ne nadje, onda dobijemo compiler error. Ova logika se primenjuje na sve metode (i instance i static). Znaci u svakom slucaju se signature metoda upisuje u bytecode tokom kompajliranja (malo redundantna recenica jer naravno da se upisuje tokom kompajliranja, ne postoji 2. trenutak kad bi nesto moglo da se upise u bytecode ako ne tokom kompajliranja).

Nakon ovoga treba doneti odluku koju konkretnu implementaciju metoda treba izvrsiti tokom runtime-a. Ovu odluku moze doneti ili compiler ili JVM zavisno od tipa metoda. Ako je u pitanju static metod onda odluku donosi kompajler na osnovu tipa reference dok u slucaju instance metoda odluku donosi JVM na osnovu tipa objekta. JVM polazi od objekta trazeci 1. metod koji zadovaljava signature iz bytecode-a. Posto se odluka o tome koji staticki metod ce se izvrsti donosi tokom kompajliranja ovo se zove compile-time ili early binding. Sa 2. strane, u slucaju instance metoda imamo runtime ili late binding (dynamic binding). Late binding je karakteristican za override-ovane metode.

U slucaju polja objekta, primenjuje se early binding na osnovu tipa reference.

Da bi videli koji metod je kompajler izabrao (set-ovao u bytecode), koristimo javap tool kojim citamo sadrzaj bytecode-a, tj. .class file-ova.

Obavezno pogledati: Obuka "Udemy/Java in depth/Section 8/115. ep." + "Method Binding" u Obuke/Java/Udemy/Java in depth/Dokumentacija + primer koda "Examples/methodBinding".

Ako imamo kod:

v1.display();

when this statement executes, the display() method is invoked in the following steps:

  1. The variable v1 is accessed.
  2. The object stored in that variable is found (following the reference).
  3. The class of the object is found (following the instance of reference).
  4. The implementation of the display method is found in the class and executed.

xh2 Immutable classes

Those class, whose object can not be modified once created, it means any modification on immutable object will result in another immutable object. Rules to make a class immutable:

  1. State of immutable object can not be modified after construction, any modification should result in new immutable object.
  2. All fields of immutable class should be final. Napomena je da final polja ne dobijaju default-nu vrednost, odnosno moramo ih rucno inicijalizovati tokom deklaracije ili u instance ili static initializer-u ili u konstruktoru.
  3. Object must be properly constructed i.e. object reference must not leak during construction process.
  4. Object should be final in order to restrict sub-class for altering immutability of parent class (to prevent subclasses from overriding methods, which can compromise immutability).
  5. Ako neko polje immutable objekta ukazuje na mutable objekat i ako to polje treba da bude dostupno i izvan objekta onda, svaki put kada neko trazi vrednost tog polja, treba mu vratiti kopiju mutable objekta na koje polje ukazuje (to prevent the risk of malicious client modifying mutable reference after getting it - reference leak).

xh2 Program vs. process vs. thread

A program is an executable file containing a set of instructions and passively stored on disk. One program can have multiple processes. For example, the Chrome browser creates a different process for every single tab.

A process means a program is in execution. When a program is loaded into the memory and becomes active, the program becomes a process. The process requires some essential resources such as registers and stack. A process can have 1 or more threads. 1 proces moze pitati OS da pokrene 2. proces. If 2 processes need to talk, they can do so by using Inter Process Communication (IPC). Many applications are designed to work this way so that if a worker process get unresponsive, it can be restarted without stopping other processes which are running different parts of the application.

A thread is the smallest unit of execution within a process.



Main differences between process and thread:

  1. Processes are usually independent, while threads exist as subsets of a process.
  2. Each process has its own memory space. Threads that belong to the same process share the same memory.

    3. Upravo to sto svaki proces ima svoj privatni deo memorije nam obezbedjuje sandboxing izmedju task-ova koji se izvrsavaju na razlicitim procesima, a to pak doprinosi i security-u, jer task u 1 procesu ne moze uticati tek tako na 2. task u 2. procesu. <- STA ZAPRAVO ZNACI SANDBOXING.

  3. A process is a heavyweight operation. It takes more time to create and terminate.
  4. Context switching is more expensive between processes.
  5. Inter Thread Communication (ITC) is faster then Inter Process Communication (IPC).

xh1 Decorator


A decorator is a Java class that is annotated javax.decorator.Decorator and that has a corresponding decorators element in the beans.xml file. A decorator bean class must also have a delegate injection point, which is annotated javax.decorator.Delegate. This injection point can be a field, a constructor parameter, or an initializer method parameter of the decorator class. Decorators are outwardly (spolja) similar to interceptors. However, they actually perform tasks complementary to those performed by interceptors. Interceptors perform cross-cutting tasks associated with method invocation and with the lifecycles of beans, but cannot perform any business logic. Decorators, on the other hand, do perform business logic by intercepting business methods of beans. This means that instead of being reusable for different kinds of applications, as are interceptors, their logic is specific to a particular application. If an application has both interceptors and decorators, the interceptors are invoked first. This means, in effect, that you cannot intercept a decorator.

xh1 Comparable


compareTo() je u istom rangu kao i equals() i hashCode() metodi. Kada se override-uje moraju se postovati odredjena pravila:

  1. compareTo() must be in consistent with equals() method e.g. if two objects are equal via equals(), there compareTo() must return 0 -> (x.compareTo(y) == 0) == (x.equals(y)). Ako ovo nije ispunjeno, potrebno je to posebno naglasiti: Note: This class has a natural ordering that is inconsistent with equals. A class whose compareTo() method imposes an order that is inconsistent with equals() will still work, but sorted collections containing elements of the class may not obey the general contract of the appropriate collection interfaces (Collection, Set, or Map). This is because the general contracts for these interfaces are defined in terms of the equals() method, but sorted collections use the equality test imposed by compareTo() in place of equals(). It is not a catastrophe if this happens, but it’s something to be aware of.

    2. For example, consider the BigDecimal class, whose compareTo() method is inconsistent with equals(). If you create a HashSet instance and add new BigDecimal("1.0") and new BigDecimal("1.00"), the set will contain 2 elements because the 2 BigDecimal instances added to the set are unequal when compared using the equals() method, because BigDecimal#equals() mehtod compares both value and scale of 2 BigDecimal instances. If, however, you perform the same procedure using a TreeSet instead of a HashSet, the set will contain only 1 element because the 2 BigDecimal instances are equal when compared using the compareTo() method.

  2. compareTo() must throw NullPointerException if current object get compared to null object as opposed to equals() which return false on such scenario.
  3. Don't use subtraction for comparing integral values because result of subtraction can overflow Integer max/min value. It is safe to use only if difference between the lowest and highest possible field values is less than or equal to Integer.MAX_VALUE ( 2 31 -1). To be safe we should use either Integer.compareTo() or logical operators (< or >) for comparison.
  4. Use Float.compareTo() or Double.compareTo() to compare floating point number since relational operator doesn't obey contract of compareTo() for floating point numbers.
  5. When comparing String using compareTo() consider case. Just like equals() consider case, compareTo() also consider case. If you want to compare regardless of case than use String.compareToIgnoreCase().

xh1 Diamond problem

A foo()

/ \

B foo() C foo()

\ /

D foo()

Ovo je jedan od razloga zasto Java ne podrzava multiple inheritance for classes.

The comment start symbol must have two asterisks to be recognized as a javadoc comment. Such a comment immediately preceding the class declaration is read as a class comment. If the comment is directly above a method signature, it is considered a method comment.

Using javadoc, several special key symbols are available for formatting the documentation. These key symbols are usually called javadoc tags and they start with the @ symbol and include:

The @author and @version tags are regularly found in class and interface comments and cannot be used in constructor, method, or field comments.

The @param and @throws tags are used with methods and constructors, whereas @return is just used with methods.

The @see tag has several different forms and may be used in any documentation comment:

The 1st simply embeds a text string with no hyperlink; the 2nd embeds a hyperlink to the specified document; the 3th links to the documentation for the isAlive method in the same class; the 4th links to the documentation for the add() method in the ArrayList class of the java.util package.

xh1 java.util.Random


Supports generation of pseudo-random values—typically, random numbers. The sequence of numbers generated is determined by a seed value, which may be passed to a constructor or set via a call to setSeed().

Two Random objects starting from the same seed will return the same sequence of values to identical calls.
